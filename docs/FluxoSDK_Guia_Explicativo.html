1. Resumo Executivo

O FluxoSDK é uma plataforma experimental de analytics comportamental. Ele tem três partes principais: um SDK JavaScript que roda no site monitorado, uma API Express que recebe e agrega eventos, e um painel Angular que transforma esses eventos em métricas visuais.

Ideia central: o site cliente carrega /assets/embed.js com uma siteKey. Esse script registra eventos de navegação, clique, formulário, scroll, performance e erro. A API valida a chave pública, grava os eventos no PostgreSQL e o dashboard consulta estatísticas por site e período.

O que o sistema já entrega

  • Cadastro e login de usuários.
  • Cadastro de sites e geração de chaves públicas pk_....
  • Coleta de eventos via script externo.
  • Armazenamento de sessões, visitas por página e eventos detalhados em JSONB.
  • Dashboard com métricas de page views, visitantes, sessões, erros, formulários, cliques problemáticos, Web Vitals, fontes, dispositivos, funis e fluxos.
  • Área administrativa para usuários root.
  • Cliente de exemplo com rotas e ações para gerar dados de teste.

5. Backend Express e PostgreSQL

Inicialização em main.ts

O backend cria um Client do PostgreSQL usando variáveis BD_HOST, BD_PORT, BD_USER, BD_PASSWORD, BD_DATABASE ou BD_NAME. Depois configura CORS aberto, JSON limitado a 256 KB, log de tráfego, arquivos estáticos em /assets e rotas em /api. 

export const pg = new Client({
  host: process.env.BD_HOST,
  port: process.env.BD_PORT ? Number(process.env.BD_PORT) : undefined,
  user: process.env.BD_USER,
  password: process.env.BD_PASSWORD,
  database: process.env.BD_DATABASE ?? process.env.BD_NAME,
  connectionTimeoutMillis: process.env.BD_TIMEOUT_MS
    ? Number(process.env.BD_TIMEOUT_MS)
    : 5000,
});

Na subida, o servidor chama connectPostgres(), tenta conectar, roda database/schema.sql e então escuta a porta definida em API_PORTA.

Padrão controller/service

Os controllers validam entrada HTTP e retornam JSON. Os services concentram regra de negócio e SQL. Isso aparece em todas as áreas: Auth, Sites, SDK e Admin.

Autenticação atual

AuthService cadastra usuário com crypt($2, gen_salt('bf')), função do pgcrypto, e faz login comparando password_hash = crypt($2, password_hash).

O frontend grava fluxosdk_user_id, nome, e-mail e flag root no localStorage. As APIs protegidas recebem o usuário via header x-user-id.

Atenção: isso funciona como autenticação simplificada para ambiente local, mas não é segurança de produção. Qualquer cliente poderia enviar outro x-user-id. O caminho natural é JWT ou sessão assinada.

Cadastro de sites

SitesService.registerSite() limpa o domínio removendo protocolo e path, grava sites e gera uma chave pública com crypto.randomUUID(). 

const cleanDomain = domain.replace(/^https?:\/\//, '').split('/')[0];
const publicKey = `pk_${crypto.randomUUID().replace(/-/g, '')}`;

Usuários root veem todos os sites. Usuários comuns só veem sites cujo sites.user_id é o seu próprio ID.

Índices

Há índices para site_key, session_id, visitor_id, page_id, event_type, occurred_at e o índice composto (site_key, event_type, occurred_at DESC). Isso sustenta as consultas do dashboard por site, tipo e período.

Fila e envio

O SDK empilha eventos em eventQueue. Ele envia quando atinge batchSize, quando o timer de flushMs dispara, ou quando uma ação pede flush imediato. Em saída de página, tenta navigator.sendBeacon e cai para fetch(..., keepalive). 

function queueEvent(event, options) {
  eventQueue.push(event);

  if (eventQueue.length >= batchSize) {
    flush(false);
    return;
  }

  if (!flushTimer) {
    flushTimer = setTimeout(function () {
      flush(false);
    }, flushMs);
  }

  if (options && options.flush) {
    flush(Boolean(options.beacon));
  }
}

8. Ingestão de Eventos e Métricas

Recebimento no backend

SdkService.receiveEvents() é o fluxo mais crítico do backend. Ele impõe MAX_EVENTS_PER_REQUEST = 100, valida obrigatórios, confirma a siteKey, confere se site e chave estão ativos, e grava tudo em transação. 

BEGIN
  upsert browser_sessions(site_key, session_id, visitor_id, user_identifier, context)

  para cada evento aceito:
    validar eventId, type, pageId, url, path, title, occurredAt
    INSERT sdk_events ... ON CONFLICT(event_id) DO NOTHING
    se page_view ou route_change:
      INSERT page_visits ... ON CONFLICT DO NOTHING
    se page_leave:
      UPDATE page_visits com left_at, duration_ms, max_scroll_depth
    se scroll_depth:
      UPDATE page_visits.max_scroll_depth
    se identify:
      UPDATE browser_sessions.user_identifier
COMMIT

9. Frontend Angular

O app frontend é uma aplicação Angular standalone. A configuração fica em app.config.ts e as rotas em app.routes.ts. 

export const appRoutes: Route[] = [
  { path: '', component: LandingComponent, canActivate: [guestGuard] },
  { path: 'auth', component: AuthComponent, canActivate: [guestGuard] },
  { path: 'dashboard', component: DashboardComponent, canActivate: [authGuard] },
  { path: '**', redirectTo: '' },
];

Guards

authGuard deixa entrar no dashboard se existir fluxosdk_user_id no localStorage. guestGuard faz o inverso: se já está logado, redireciona para /dashboard.

Landing

LandingComponent monta uma home visual com mock de dashboard em celular. Ele simula eventos vivos por timer, alterna KPIs e direciona para login. É uma peça demonstrativa, não participa diretamente da coleta real.

Autenticação

AuthComponent alterna entre login e cadastro com sinais. Ele valida nome, e-mail, senha e confirmação; chama AuthService; salva dados do usuário no localStorage; e navega para /dashboard. 

localStorage.setItem('fluxosdk_user_id', user.id);
localStorage.setItem('fluxosdk_user_name', user.name || '');
localStorage.setItem('fluxosdk_user_email', user.email);
localStorage.setItem('fluxosdk_user_is_root', String(Boolean(user.is_root)));

Dashboard

DashboardComponent é a tela principal. Ele mantém estado com signals, busca sites, seleciona a primeira siteKey, busca eventos e métricas, renderiza gráficos Chart.js e atualiza a cada 8 segundos. 

private async fetchDashboardData() {
  const siteKey = this.selectedSiteKey();
  if (!siteKey) {
    this.liveEvents.set([]);
    this.stats.set(null);
    return;
  }

  const [events, stats] = await Promise.all([
    this.sdkEventsService.getRecentEvents(siteKey, 40),
    this.sdkEventsService.getStats(siteKey, this.rangeHours()),
  ]);

  this.liveEvents.set(events);
  this.stats.set(stats);
  this.renderCharts();
}

Abas do dashboard

  • Dashboard: KPIs, gráficos, listas analíticas, performance, funis e fluxo.
  • Eventos: timeline dos eventos recentes.
  • Sessões: dispositivos, fontes, engajamento e contexto técnico.
  • Sites: cadastro, listagem, siteKey e exclusão.
  • Admin: visível só para root, com usuários, sistemas e totais globais.

12. Testes e Qualidade

O repositório tem estrutura de testes criada pelo Nx, mas parte dos testes ainda não foi atualizada para a aplicação real.

13. Pontos de Atenção e Melhorias

Tema Observação Sugestão

CORS Access-Control-Allow-Origin: *. Restringir por domínio cadastrado ou por ambientes.

Porta da API API_PORTA não tem fallback no código. Usar fallback 3000 e validar env obrigatória na subida.

14. Conclusão

O FluxoSDK já tem uma linha de produto funcional: um script de analytics completo, uma API que persiste eventos e calcula métricas ricas, e um dashboard Angular capaz de contar a história do comportamento do usuário. A espinha dorsal técnica está bem definida.

Os maiores próximos ganhos estão em segurança de autenticação, configuração por ambiente, revisão de encoding/textos, testes reais e controles de privacidade/volume. Com esses ajustes, o projeto deixa de ser apenas um laboratório sólido e se aproxima de uma base pronta para uso controlado.
